# EyeClaw 安全与隐私说明 v0.1

日期：2026-05-26

## 1. 说明目的

EyeClaw 会记录用户在软件中的真实操作，并可能保存录屏、截图、页面文本、点击事件和输入事件。因此，安全与隐私边界必须在测试和部署前讲清楚。

本说明面向企业测试负责人、IT 安全负责人、业务负责人和 EyeClaw 操作员，用于明确：

- EyeClaw 会采集什么。
- EyeClaw 不应采集什么。
- 数据保存在什么位置。
- 哪些数据可能发送给模型。
- 遇到密码、验证码、扫码登录等敏感场景如何处理。
- 企业私有化部署时应采用哪些控制措施。

## 2. 基本原则

EyeClaw 的安全设计应遵守以下原则：

1. 用户明确开始后才采集。默认不后台监听用户所有操作。
2. 最小必要采集。只采集理解业务流程所需的数据。
3. 本地优先。个人版和 PoC 阶段默认在本机保存采集材料。
4. 人工接管优先。密码、验证码、扫码登录、二次确认等环节不承诺自动完成。
5. 可审计。录屏、截图、事件日志、模型调用和执行日志应能追踪来源。
6. 可删除。测试材料应有明确保存周期和删除方式。
7. 不把演示等同授权。录到某个动作，不代表系统获得长期自动执行权限。

## 3. 数据采集范围

### 3.1 浏览器监听事件

当前浏览器扩展可能采集：

- 页面 URL。
- 页面标题。
- 点击事件。
- 输入事件。
- 选择和变更事件。
- 滚动事件。
- 页面跳转和标签页变化。
- 目标元素文本。
- 目标元素标签和类型。
- 候选选择器。
- 事件时间戳。
- 关键事件附近的截图。

这些数据用于生成事件时间轴、识别操作步骤和引导关键帧抽取。

### 3.2 录屏和截图

当前系统可能采集：

- 当前浏览器标签页录屏。
- 关键事件附近截图。
- 从录屏中抽取的关键帧。

录屏和截图用于让多模态模型理解页面状态，尤其是监听事件无法表达的视觉上下文，例如弹窗、菜单展开、列表内容、按钮位置和页面布局。

### 3.3 技能和执行日志

系统可能保存：

- 技能名称。
- 技能说明。
- 用户填写的业务目标。
- 自动生成的 SOP。
- 结构化步骤计划。
- 执行日志。
- 失败原因。
- 人工修正记录。

这些材料构成可复用技能资产，应按企业内部知识资产管理。

## 4. 不建议采集或自动处理的数据

第一阶段测试不建议采集：

- 真实密码。
- 短信验证码。
- 动态口令。
- 扫码登录二维码细节。
- 身份证号、银行卡号、医疗记录等高敏个人信息。
- 商业合同、客户名单、财务明细等未经授权的敏感材料。
- 生产环境不可逆操作的确认页面。

如果流程中必须经过这些环节，建议做法是：

- 由人手动完成。
- 在技能中标记为人工接管点。
- 不把敏感值保存为可回放输入。
- 不让模型猜测或生成敏感凭据。

## 5. 本地保存位置

当前原型默认在项目本地保存运行材料，例如：

- 监听截图：`artifacts/listener_frames/`
- 会话录屏：`artifacts/session_recordings/`
- 技能库和分析结果：以本地项目实现为准
- 浏览器测试配置：`.browser/`

企业 PoC 前应确认：

- 这些目录是否允许保存测试数据。
- 是否需要把目录放到加密磁盘或企业指定路径。
- 是否需要测试后统一删除。
- 是否需要禁止提交到 Git。

## 6. 模型调用边界

EyeClaw 的分析可能调用多模态模型和语言模型。根据配置不同，模型可能是：

- 外部云模型。
- 企业内网模型。
- 本地模型。

可能发送给模型的数据包括：

- 关键截图或关键帧。
- 页面标题和 URL。
- 元素文本。
- 用户填写的任务目标。
- 事件时间轴摘要。
- 已生成的步骤草稿。

企业测试前必须明确：

1. 是否允许截图出网。
2. 是否允许页面文本出网。
3. 是否允许用户输入内容出网。
4. 是否允许录屏关键帧出网。
5. 模型服务商是否会保留请求数据。
6. 是否需要改用企业内网模型。

如果无法确认这些问题，建议采用企业内网模型或本地模型进行 PoC。

## 7. 密码、验证码和人工接管

EyeClaw 对敏感认证环节的原则是记录流程位置，不记录凭据本身。

### 7.1 密码输入

建议：

- 使用测试账号。
- 避免在录屏中展示正式密码。
- 不把密码保存为技能步骤的明文输入。
- 对密码框输入事件做脱敏处理。

### 7.2 验证码和短信码

建议：

- 人工完成。
- 标记为人工接管点。
- 不保存验证码内容。
- 不训练系统自动绕过验证码。

### 7.3 扫码登录

建议：

- 人工扫码。
- 系统记录“需要扫码登录”这个状态。
- 不尝试复用二维码。
- 不承诺绕过二次认证。

### 7.4 二次确认和高风险提交

建议：

- 在 PoC 阶段使用测试环境。
- 对删除、付款、发布、审批通过等动作要求人工确认。
- 技能中明确标记风险动作。

## 8. 数据脱敏建议

企业版应逐步支持以下脱敏能力：

- 密码字段不保存明文。
- 输入框按字段类型打码。
- URL 中的 token、session、ticket 参数脱敏。
- 截图中指定区域模糊化。
- 录屏片段裁剪。
- 页面文本中的手机号、身份证号、邮箱、银行卡号自动替换。
- 模型调用前生成脱敏版本上下文。

PoC 阶段如果尚未实现自动脱敏，应通过测试数据和人工流程控制降低风险。

## 9. 权限和审计

团队版和企业版建议具备：

- 用户登录和身份识别。
- 技能创建权限。
- 技能执行权限。
- 录屏查看权限。
- 技能发布审批。
- 执行日志审计。
- 敏感技能访问控制。
- 技能版本记录。

关键问题是区分三类权限：

1. 能看演示材料。
2. 能编辑技能。
3. 能执行技能。

这三类权限不应默认相同。

## 10. 部署模式

### 10.1 本地个人模式

适合个人试用和早期开发：

- EyeClaw 服务运行在用户本机。
- 数据保存在本地项目目录。
- 浏览器扩展连接本地服务。
- 用户自行配置模型。

优点是简单，风险是缺少统一治理。

### 10.2 企业 PoC 模式

适合小范围测试：

- 指定测试机。
- 使用测试账号和脱敏数据。
- 明确录屏保存目录。
- 明确模型调用方式。
- 测试后统一清理材料。

优点是可控，适合验证价值。

### 10.3 企业私有化模式

适合正式部署：

- 服务部署在企业内网。
- 模型使用企业内网模型或本地模型。
- 技能库由企业统一管理。
- 接入企业身份系统。
- 日志纳入审计系统。
- 敏感数据按企业制度保存和删除。

优点是安全边界清晰，适合长期使用。

## 11. 风险清单

| 风险 | 说明 | 建议控制 |
|---|---|---|
| 敏感信息进入录屏 | 页面中出现密码、客户信息或财务数据 | 使用测试数据，截图脱敏，限制保存 |
| 模型调用出网 | 截图或页面文本发送给外部模型 | 使用内网模型，调用前脱敏 |
| 技能误执行 | 技能在错误页面或生产环境执行 | 起始页面校验，高风险动作人工确认 |
| 权限扩大 | 会演示的人不一定应授权系统执行 | 技能发布审批，执行权限分离 |
| 页面变化导致误点 | UI 变化后定位错误 | 执行前观察，失败停止，人工确认 |
| 测试材料遗留 | PoC 后录屏和日志未清理 | 设置保存周期和删除流程 |
| 误解人工判断 | 模型把人的业务判断误写成固定规则 | 在技能中标记不确定性和人工判断点 |

## 13. 建议默认策略

在没有特殊企业授权前，建议采用以下默认策略：

- 只在测试环境运行。
- 只使用测试账号。
- 只使用脱敏数据。
- 默认本地保存材料。
- 默认不自动处理密码、验证码、扫码登录。
- 默认高风险提交需要人工确认。
- 默认外部模型不接收高敏截图。
- 测试结束后清理录屏、截图和日志。

## 14. 结论

EyeClaw 的价值来自真实操作数据，但风险也来自真实操作数据。企业落地时，不能只关注能不能学会流程，还要关注采集边界、模型调用边界、权限边界和执行边界。

合理的安全策略不是禁止采集，而是让采集可见、可控、可审计、可删除。这样 EyeClaw 才能从 Demo 走向企业可用的技能学习系统。
