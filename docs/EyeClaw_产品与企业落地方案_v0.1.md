# EyeClaw 产品与企业落地方案 v0.1

日期：2026-05-26

## 1. 一句话定位

EyeClaw 是一个“能看、能听、能学会软件操作”的本地智能体原型：它通过监听用户在软件中的真实操作，并结合录屏关键帧与多模态模型理解，把一次业务演示沉淀为可读 SOP、可回放步骤和可复用技能。

更直白地说，EyeClaw 不是让业务人员写提示词、写脚本或配置复杂自动化流程，而是让他们像教新人一样演示一遍，系统从“过程 + 结果”中学习。

## 2. 我对 EyeClaw 的理解

EyeClaw 的核心价值不是“录屏转文字”，也不是传统 RPA 的另一种 UI 自动化实现。它更像是软件智能体的学习层：把一线业务人员在真实软件中的隐性经验，转成机器可以理解、验证、执行和复用的技能资产。

这个项目背后的几个判断很重要：

1. 语言和结构化数据优先。点击、输入、DOM、页面文本、URL、元素标签等监听数据，比单纯视觉帧更稳定，也更容易沉淀为技能。
2. 视觉不是替代监听，而是补足监听。录屏和关键帧用于理解页面状态、校验动作语义、发现监听数据无法表达的上下文。
3. 过程学习优于只看结果。很多业务技能无法从最终页面或最终文件反推出来，必须记录人是如何一步步完成的。
4. 软件本身就是智能体训练场。软件有状态、有反馈、有错误、有可重复任务，天然适合智能体学习和强化。
5. 对一线业务人员来说，演示比写提示词更自然。长时间演示会暴露大量无法被语言完整描述的细节，例如等待时机、人工判断、异常分支和业务习惯。

在这个理解下，EyeClaw 可以被定位为 OpenClaw 的“有眼睛升级版”：OpenClaw 是个人 AI 助手和技能运行环境，EyeClaw 让这个助手具备观察真实软件操作、从演示中学习技能的能力。`Video2Skills` 更适合作为 EyeClaw 的核心技术模块或论文方法名，用来描述“从操作视频和监听轨迹生成技能”的技术链路。

## 3. 要解决的问题

企业和个人在软件使用中有大量重复但难以自动化的流程：

- 老员工知道怎么做，但新人只能靠口头教学或零散文档学习。
- SOP 经常落后于真实系统界面，维护成本高。
- RPA 需要专业人员配置，遇到页面变化容易失效。
- 浏览器 Agent 可以操作页面，但缺少来自业务人员的稳定技能来源。
- 业务人员会演示流程，却不会把流程写成高质量提示词、脚本或自动化任务。
- 很多流程包含人工判断、登录验证、异常处理，不能简单机械复刻。

EyeClaw 的目标是把“演示”变成一种低门槛的技能生产方式。

## 4. 目标用户

### 4.1 个人用户

适合经常在网页和软件中重复操作的人，例如：

- 运营人员：后台配置、内容发布、数据导出。
- 自媒体/独立开发者：网页表单、素材处理、发布流程。
- 研究人员：论文检索、数据下载、网页整理。
- 普通知识工作者：把自己常做的浏览器任务保存成个人技能。

个人版重点是本地运行、低门槛安装、快速生成 SOP 和可复用流程。

### 4.2 团队用户

适合有多个成员共享流程的团队，例如：

- 客服团队：工单处理、账号检查、标准回复流程。
- 财务团队：报表下载、发票核验、系统录入。
- 销售运营：CRM 更新、线索处理、数据核对。
- 行政/人事：审批系统、入离职流程、信息维护。

团队版重点是技能库、审核、版本管理和复用。

### 4.3 企业用户

适合存在大量内部系统和复杂流程的企业，例如：

- 需要把一线人员经验沉淀为组织资产。
- 内部系统多、接口少，传统 API 集成成本高。
- RPA 已经使用但维护成本高。
- 希望在不改造原系统的情况下引入智能体能力。

企业版重点是私有化部署、数据安全、权限审计、脱敏、本地模型或内网模型接入。

## 5. 产品闭环

EyeClaw 的最小闭环是：

```text
创建技能目标
  -> 开始监听和录屏
  -> 用户真实演示一次业务流程
  -> 停止演示
  -> 监听器生成事件时间轴
  -> 录屏按事件时间轴抽取关键帧
  -> 多模态模型理解页面状态和操作语义
  -> 生成 SOP、回放计划和技能草稿
  -> 人工确认/编辑
  -> 保存到技能库
  -> 在原软件上执行或辅助执行
```

这个闭环的关键不是一次性自动化成功，而是让技能可以被审阅、修改、测试、复用和持续更新。

## 6. 当前能力

当前 Demo 已经具备以下基础能力：

- 浏览器扩展监听点击、输入、滚动、页面跳转、标签页变化等事件。
- 监听会话有独立 `session_id`，便于把事件、截图、录屏归到同一次演示。
- 监听开始和录屏开始同步，停止后上传会话录屏。
- 事件时间轴可以引导关键帧抽取，避免只做平均抽帧。
- 前端控制台支持创建技能、开始演示、停止演示、分析演示过程、保存技能和执行技能。
- 回放执行使用 Playwright 和现有浏览器状态，支持人工登录后的流程继续。
- 已有失败恢复策略，例如重新观察页面、宽松定位、滚动后点击、强制点击、DOM 点击、候选项匹配。
- 已有新手教程和操作手册，适合继续包装为企业 PoC 指南。

## 7. 技术架构

EyeClaw 可以分成五层：

### 7.1 采集层

采集层负责记录真实演示过程：

- 浏览器扩展采集点击、输入、滚动、导航、页面标题、URL、元素文本、选择器等监听数据。
- 录屏模块记录用户操作过程。
- 关键事件附近保存截图。
- 控制台事件会被过滤，避免把 EyeClaw 自身操作混入目标网站流程。

### 7.2 对齐层

对齐层把多种数据源放到同一条时间线上：

- 监听事件时间戳。
- 录屏时间轴。
- 截图时间点。
- 页面状态变化。

这层决定了 EyeClaw 和普通录屏总结工具的差异：抽帧不是平均抽帧，而是由真实操作事件引导。

### 7.3 理解层

理解层调用多模态模型和语言模型，把原始轨迹转成结构化步骤：

- 多模态模型理解关键帧中的页面状态和可见 UI。
- 语言模型结合监听事件，推断用户意图和步骤语义。
- 系统生成可读 SOP、结构化动作列表和不确定性说明。

### 7.4 技能层

技能层把一次演示变成可复用资产：

- 技能名称、目标、说明。
- 步骤列表。
- 页面定位策略。
- 人工接管点。
- 成功条件。
- 失败恢复建议。
- 版本和来源记录。

后续可以进一步映射到 OpenClaw 的 workspace skills 或 bundle plugin。

### 7.5 执行层

执行层负责在原软件上应用技能：

- 使用 Playwright 连接真实浏览器。
- 复用用户已登录状态。
- 按技能步骤执行。
- 页面变化时尝试自适应定位。
- 遇到扫码、验证码、二次确认时交给人类接管。
- 记录执行日志，反哺技能修订。

## 8. 和现有方案的区别

### 8.1 相比传统 RPA

传统 RPA 强调配置流程和稳定重放，通常需要专人维护。EyeClaw 更强调从业务人员演示中学习，先生成可审阅技能，再逐步提高可执行性。

RPA 更像“把流程写出来再执行”，EyeClaw 更像“看人做一遍后形成技能”。

### 8.2 相比浏览器 Agent

浏览器 Agent 通常依赖实时推理，任务每次都重新观察和规划。EyeClaw 关注技能沉淀：把一次成功演示变成可复用经验，让后续执行不完全依赖临场推理。

### 8.3 相比录屏教程

录屏教程主要服务人类学习。EyeClaw 会把录屏、监听事件和页面状态转成结构化动作，让机器也能使用这些知识。

### 8.4 相比流程挖掘

流程挖掘通常依赖系统日志和业务事件。EyeClaw 面向没有统一日志、没有 API、只有真实软件界面的场景，通过浏览器监听和视觉理解捕捉流程。

## 9. 典型场景

### 9.1 业务后台配置

运营人员演示一次后台配置流程，例如创建活动、设置规则、上传素材、发布页面。EyeClaw 生成 SOP 和技能，下次新人可以按 SOP 执行，也可以让系统辅助回放。

### 9.2 报表下载与录入

财务人员演示从多个系统下载报表、筛选条件、保存文件、录入台账。EyeClaw 记录筛选条件、人工判断点和下载动作，沉淀为可审阅技能。

### 9.3 客服工单处理

客服主管演示如何判断工单类型、查询用户信息、选择处理模板、提交结果。EyeClaw 可以沉淀标准操作流程，并标记需要人工判断的环节。

### 9.4 企业内训

老员工不用写长文档，只需要演示一遍完整流程。EyeClaw 输出新人教程、步骤清单和可回放任务，降低培训成本。

### 9.5 个人自动化

个人用户把常用网页任务录制成技能，例如查资料、整理网页信息、批量提交表单、重复下载材料。

## 10. 企业 PoC 测试方案

### 10.1 测试目标

验证 EyeClaw 是否能把真实业务演示转成可读、可复用、部分可执行的技能资产。

PoC 不应只看“能否一次全自动跑通”，而应同时评估：

- 是否准确理解业务步骤。
- 是否减少 SOP 编写成本。
- 是否能定位人工接管点。
- 是否能在相似页面状态下复用。
- 是否能为后续自动化提供稳定技能草稿。

### 10.2 测试流程选择

建议选择 3 到 5 个流程，分三类：

1. 简单流程：5 到 8 步，无复杂登录，无强动态页面。
2. 中等流程：10 到 20 步，包含筛选、表单、下载、页面跳转。
3. 复杂流程：包含人工判断、验证码、二次确认、动态菜单或异常分支。

每个流程都由真实业务人员演示，不由技术人员代替。

### 10.3 测试环境

建议准备：

- 一台 Windows 测试机。
- Edge 或 Chrome。
- 企业测试账号。
- 可脱敏的测试数据。
- 允许本地录屏和本地保存事件日志。
- 如果使用外部模型，需要明确哪些数据会出网。
- 如果企业要求内网闭环，PoC 应接入企业内网模型或本地模型。

### 10.4 交付物

每个测试流程至少输出：

- 原始监听事件。
- 会话录屏或关键帧。
- 自动生成 SOP。
- 结构化步骤计划。
- 可执行技能草稿。
- 执行日志。
- 人工修正记录。
- 成功和失败原因分析。

### 10.5 验收指标

建议第一阶段使用以下指标：

| 指标 | 目标 |
|---|---|
| SOP 可读性 | 业务人员能看懂并指出问题 |
| 步骤完整率 | 关键业务步骤不漏掉 |
| 操作语义准确率 | 按钮、筛选项、输入值、页面跳转基本准确 |
| 人工修正成本 | 修正技能比从零写 SOP 或脚本更快 |
| 回放成功率 | 简单流程可稳定回放，中等流程可在人工接管下完成 |
| 人工接管识别 | 登录、验证码、二次确认等环节被明确标记 |
| 数据安全 | 敏感输入和录屏材料有保存边界和脱敏策略 |

## 11. 隐私与安全边界

EyeClaw 会接触用户真实软件操作，因此安全边界必须前置说明。

当前建议：

- 默认本地运行。
- 默认用户明确点击开始后才采集。
- 不采集浏览器内部页面，例如 `edge://`、`chrome://`。
- 对密码、验证码、扫码登录等场景不承诺自动完成。
- 人工接管点需要被记录为“需要人处理”，而不是让模型猜测。
- 录屏和截图默认保存在本地测试目录。
- 企业版应支持敏感字段脱敏、录屏片段裁剪、日志审计和数据保留周期配置。
- 外部模型调用前必须明确发送的数据类型，包括截图、页面文本、事件日志和用户输入。

企业部署时，建议优先采用“本地采集 + 企业内网模型 + 企业内网技能库”的方案。

## 13. 技能生命周期

EyeClaw 的长期价值在于技能不是一次性产物，而是持续演化的组织资产。

建议生命周期如下：

```text
演示采集
  -> 自动分析
  -> 技能草稿
  -> 人工确认
  -> 测试回放
  -> 发布入库
  -> 复用执行
  -> 执行日志反馈
  -> 技能修订
  -> 新版本发布
```

这个生命周期也适合未来论文中的系统框架描述。

## 14. 与 Ctx2Skill、CLI-Anything、OpenClaw 的关系

### 14.1 Ctx2Skill

Ctx2Skill 关注从上下文中生成可复用技能。EyeClaw 可以借鉴它的“技能自生成”机制，但 EyeClaw 的上下文不是静态文本，而是真实软件操作过程：监听事件、录屏关键帧、页面状态和执行结果。

### 14.2 CLI-Anything

CLI-Anything 关注把软件能力封装成 Agent 可调用的 CLI。EyeClaw 可以借鉴它把软件暴露给 Agent 的思路。不同的是，EyeClaw 先从人的演示中发现技能，再决定是否生成 CLI、脚本、Playwright 步骤或 OpenClaw skill。

### 14.3 OpenClaw

OpenClaw 可以作为 EyeClaw 技能的运行环境和个人 AI 助手底座。EyeClaw 更像 OpenClaw 的“观察与学习模块”：让 OpenClaw 不只会调用已有技能，还能从用户演示中学习新技能。

建议近期先保持 EyeClaw 独立迭代。等技能格式稳定后，再考虑：

1. 将 EyeClaw 生成的技能映射到 OpenClaw workspace skills。
2. 把 EyeClaw 打包成 OpenClaw bundle plugin。
3. 如需深度运行时能力，再做 OpenClaw code plugin。

## 15. 命名建议

建议采用：

- 总产品名：EyeClaw。
- 核心技术名：Video2Skills。
- 生态关系：EyeClaw as an observing and learning extension of OpenClaw。

原因：

- EyeClaw 延续 OpenClaw 语义，表达“有眼睛、能观察的助手”。
- Video2Skills 适合作为论文标题、方法模块或技术子系统，表达清晰，但作为产品名略窄，因为 EyeClaw 不只依赖视频，还依赖监听、DOM、时间轴和执行日志。
- 如果未来扩展到桌面软件，EyeClaw 仍然成立；Video2Skills 则容易被理解成只处理视频。

## 16. 路线图

### 阶段 1：可演示闭环

目标：让个人或企业测试人员能完成一次端到端演示。

重点：

- 一键启动。
- 扩展安装简化。
- 录制、分析、保存、执行链路稳定。
- 示例视频和示例技能。
- 基础教程完善。

### 阶段 2：企业 PoC

目标：在企业测试环境验证真实业务流程。

重点：

- PoC 指南。
- 测试流程选择模板。
- 验收指标。
- 数据安全说明。
- 失败案例记录。
- 技能人工编辑体验。

### 阶段 3：技能资产化

目标：让技能可维护、可审计、可复用。

重点：

- 技能版本管理。
- 技能测试集。
- 执行日志反哺。
- 失效检测。
- 团队共享技能库。

### 阶段 4：OpenClaw 集成

目标：让 EyeClaw 生成的技能进入 OpenClaw 生态。

重点：

- OpenClaw skill 映射。
- Bundle plugin。
- 权限和工具声明。
- 与 OpenClaw 助手交互。

### 阶段 5：从浏览器扩展到任意软件

目标：参考 CLI-Anything，把 EyeClaw 从浏览器迁移到更多软件。

重点：

- 应用窗口识别。
- 操作事件捕获。
- 软件特定适配器。
- CLI/API/GUI 混合执行。
- 跨软件流程技能。

## 18. 结论

EyeClaw 的机会不在于再做一个能点网页的 Agent，而在于建立一条新的技能生产路径：让业务人员通过演示把经验交给系统，系统再把这些经验沉淀为可审阅、可测试、可复用、可集成的技能。

如果这条路径跑通，EyeClaw 可以同时服务个人效率、团队知识沉淀和企业智能体落地。它也可以成为 OpenClaw 的关键升级方向：让助手不只会执行已有能力，还能持续观察、学习和积累新的软件技能。
