# EyeClaw 企业 PoC 测试指南 v0.1

日期：2026-05-26

## 1. 测试目标

本指南用于在企业测试环境中验证 EyeClaw 是否能把真实业务人员的浏览器操作演示，转化为可审阅、可复用、可部分执行的技能资产。

PoC 的重点不是证明 EyeClaw 已经可以替代所有人工操作，而是验证这条路径是否成立：

```text
业务人员演示一次真实流程
  -> EyeClaw 采集监听事件和录屏关键帧
  -> 系统生成 SOP、结构化步骤和技能草稿
  -> 业务人员确认和修正
  -> 技能在相似环境中辅助执行或回放
```

测试结论应回答三个问题：

1. EyeClaw 是否能正确理解业务流程的关键步骤。
2. EyeClaw 是否比人工从零写 SOP 或自动化脚本更省时间。
3. EyeClaw 生成的技能是否具备后续复用和持续改进价值。

## 2. 推荐测试范围

### 2.1 适合第一阶段测试的流程

优先选择以下类型：

- 浏览器内完成的业务流程。
- 操作步骤在 5 到 25 步之间。
- 包含点击、输入、筛选、搜索、下载、提交、页面跳转等常见动作。
- 可以使用测试账号和脱敏数据。
- 业务人员能清楚判断流程是否正确。
- 流程结果可验证，例如文件下载、表单提交、列表筛选结果、详情页打开。

### 2.2 暂不建议第一阶段测试的流程

以下流程可以记录，但不建议作为第一批验收对象：

- 高度依赖验证码、短信码、扫码、硬件 UKey 的流程。
- 涉及真实资金、真实生产变更或不可逆提交的流程。
- 页面每次变化很大、没有稳定业务规则的流程。
- 含有大量敏感个人信息且暂未配置脱敏策略的流程。
- 需要跨多个桌面软件联动，但当前没有软件适配器的流程。

## 3. 角色分工

| 角色 | 职责 |
|---|---|
| 业务演示人 | 按真实习惯演示流程，判断生成步骤是否符合业务 |
| 企业测试负责人 | 选择流程、准备账号、组织验收、确认结论 |
| EyeClaw 操作员 | 启动系统、加载扩展、记录测试结果、导出材料 |
| 安全/IT 负责人 | 确认测试环境、数据边界、模型调用方式 |
| 观察员 | 记录人工修正点、失败原因、改进建议 |

建议每次测试至少包含业务演示人、EyeClaw 操作员和测试负责人。

## 4. 测试前准备

### 4.1 环境准备

- Windows 测试机。
- Edge 或 Chrome 浏览器。
- EyeClaw 本地服务。
- EyeClaw 浏览器监听扩展。
- 可访问的企业测试系统。
- 测试账号和测试数据。
- 明确的模型调用方式：外部模型、企业内网模型或本地模型。

### 4.2 数据准备

每个流程应准备：

- 流程名称。
- 业务目标。
- 起始页面。
- 测试账号。
- 输入样例。
- 期望结果。
- 是否允许录屏。
- 是否允许截图。
- 是否允许把截图和页面文本发送给模型。

### 4.3 风险确认

测试前需要确认：

- 不使用真实资金和不可逆生产操作。
- 不在录屏中展示正式密码、真实验证码、真实客户敏感信息。
- 如果必须出现敏感信息，应先使用测试数据或脱敏数据。
- 明确录屏、截图和事件日志保存路径。
- 明确测试材料保存周期和删除方式。

## 5. 测试流程分级

建议每次 PoC 至少覆盖三类流程。

| 类型 | 建议数量 | 特征 | 验证重点 |
|---|---:|---|---|
| 简单流程 | 1 到 2 个 | 5 到 8 步，无复杂登录 | 基础采集和 SOP 生成 |
| 中等流程 | 1 到 2 个 | 10 到 20 步，包含筛选、输入、下载 | 步骤完整性和技能复用 |
| 复杂流程 | 1 个 | 有人工判断、动态菜单、二次确认 | 人工接管和异常记录 |

第一轮不追求数量，优先把 3 个流程测透。

## 6. 单个流程测试步骤

### 6.1 建立测试记录

填写：

- 流程编号。
- 流程名称。
- 演示人。
- 测试系统。
- 起始 URL。
- 期望结果。
- 数据敏感等级。

### 6.2 开始演示

1. 启动 EyeClaw 本地服务。
2. 打开 EyeClaw 控制台。
3. 确认浏览器扩展已连接。
4. 填写技能名称、技能说明和业务目标。
5. 点击开始演示。
6. 业务演示人按真实流程操作。
7. 遇到登录、验证码、扫码、确认弹窗时，由人手动完成。
8. 流程完成后点击停止演示。

### 6.3 分析和确认

1. 点击分析演示过程。
2. 检查生成的步骤。
3. 标记错误、遗漏和多余步骤。
4. 确认是否正确识别人工接管点。
5. 保存技能草稿。

### 6.4 回放或辅助执行

1. 在测试浏览器中准备相似起始页面。
2. 选择已保存技能。
3. 执行任务。
4. 记录成功步骤、失败步骤和人工介入位置。
5. 如果失败，记录页面状态、错误原因和是否可通过人工修正继续。

## 7. 交付物

每个流程测试结束后，应保留以下材料：

- 流程测试记录表。
- 原始监听事件摘要。
- 关键截图或关键帧。
- 会话录屏路径，若企业允许保存。
- 自动生成 SOP。
- 结构化步骤计划。
- 技能草稿。
- 回放执行日志。
- 人工修正记录。
- 测试结论。

## 8. 验收指标

### 8.1 建议指标

| 指标 | 说明 | 通过参考 |
|---|---|---|
| 步骤完整率 | 关键业务步骤是否被覆盖 | 简单流程不漏关键步骤，中等流程漏项可人工补齐 |
| 操作语义准确率 | 按钮、字段、输入值、筛选项是否理解正确 | 主要动作准确，允许少量文案不精确 |
| SOP 可读性 | 业务人员能否按生成 SOP 理解流程 | 业务人员能指出问题并确认主线 |
| 技能可编辑性 | 生成结果是否便于人工修正 | 修正成本低于从零写流程 |
| 回放可用性 | 是否能在相似页面状态下执行 | 简单流程可回放，中等流程可辅助执行 |
| 人工接管识别 | 登录、验证码、人工判断是否被标记 | 不把人工环节误承诺为自动化 |
| 数据边界可控 | 录屏、截图、模型调用是否符合测试要求 | 安全负责人认可 |

### 8.2 分级结论

建议把每个流程评为四档：

| 等级 | 含义 |
|---|---|
| A | SOP 准确，技能可直接或基本直接复用 |
| B | 主流程正确，需要少量人工修正后可用 |
| C | 能提供有价值草稿，但还不能稳定执行 |
| D | 理解失败或采集不足，需要重新录制或改造流程 |

PoC 第一阶段的合理目标是：简单流程达到 A 或 B，中等流程达到 B 或 C，复杂流程能清楚记录失败原因和人工接管点。

## 9. 失败类型记录

失败不要只写“未成功”，要分类记录：

- 采集失败：扩展未连接、监听未启动、录屏失败。
- 时间轴失败：操作太快、关键帧未捕捉到页面状态。
- 理解失败：模型误解按钮、字段、业务含义。
- 执行失败：定位不到元素、页面变化、弹窗遮挡、滚动位置不同。
- 权限失败：账号权限不足、登录过期、验证码阻断。
- 数据失败：测试数据不存在、列表为空、筛选条件不匹配。
- 安全限制：敏感数据不可发送模型、录屏不允许保存。

每类失败都应记录是否可以通过重新演示、人工修正、增加提示、增加适配器或调整安全策略解决。

## 11. 推荐时间安排

### 第 1 天：环境和流程确认

- 安装 EyeClaw。
- 加载浏览器扩展。
- 确认模型调用。
- 选定 3 到 5 个测试流程。
- 跑通一个简单示例。

### 第 2 天：正式录制和分析

- 每个流程由业务人员演示。
- EyeClaw 生成 SOP 和技能草稿。
- 业务人员做第一次确认。
- 记录错误和修正成本。

### 第 3 天：回放和复盘

- 尝试技能回放或辅助执行。
- 统计成功率和失败类型。
- 形成 PoC 结论。
- 决定下一轮测试范围。

## 12. PoC 通过标准

建议企业 PoC 的通过标准不是“所有流程全自动成功”，而是满足以下条件：

1. 至少 2 个流程能生成业务人员认可的 SOP。
2. 至少 1 个简单流程能在相似环境中回放或辅助执行。
3. 中等或复杂流程能清楚识别人工接管点和失败原因。
4. 从演示到技能草稿的时间明显低于人工写 SOP 或脚本。
5. 安全负责人认可测试数据边界和模型调用方式。

如果达到以上标准，可以进入第二阶段：技能编辑体验、团队技能库、企业私有化部署和 OpenClaw 集成验证。
